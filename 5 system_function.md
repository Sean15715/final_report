# ScoutAgent 前端系统功能说明

> 整套前端基于 React 18 + TypeScript + Vite + React Router 7，由左侧固定的图标导航栏串联起 5 个核心页面。
> 所有页面共享 JWT 鉴权、流式（SSE）通信与统一的 Markdown 渲染体系。

| # | 页面 | 路由 | 一句话定位 |
|---|------|------|------------|
| 1 | 企业与岗位管理 | `/enterprises` | 招聘信息的"源头"：维护客户、JD 与投递关系 |
| 2 | 候选人管理 | `/candidates` | 简历的"档案库"：PDF → 结构化 Profile |
| 3 | 批量化评估 | `/assess` | 全局视角的"流水线评估"：批量打分与筛选 |
| 4 | **深度评估 Agent** | `/assess/assistant` | **基于 Agent 的"对话式深评"：多轮、可追溯、有记忆** |
| 5 | Eval 系统 | `/eval` | 离线评测中心：跑评测、看指标、做回归 |

---

## 一、企业与岗位管理（EnterpriseWorkspace）

业务定位：**招聘流程的入口**。所有评估的上游数据（企业、JD、投递）都在此创建与维护。

**三栏式工作台布局**：

- **左栏 · 企业列表**：搜索、新建、编辑、删除企业；支持邮箱/电话/描述等基础字段。
- **中栏 · 岗位列表**：进入企业后展示其下所有 JD，支持新建岗位、状态切换（草稿 / 开放 / 关闭）。
- **右栏 · 岗位详情（Tab 切换）**：
  - **岗位详情**：JD 标题、描述、要求文本（`requirements_text`）等结构化字段。
  - **投递列表**：在该 JD 下挂载候选人，支持上传 PDF 简历直接生成投递记录；展示投递状态徽章（已投递 / 筛选中 / 面试中 / 通过 / 拒绝 / 录用）。
  - **偏好记忆**：展示由 Agent 自动沉淀的「企业级偏好」与「岗位级偏好」（如"硬性要求 5 年以上 Go 经验"），并允许人工删除。

> 这一页是后续所有评估的"上下文池"——后端记忆系统会持续向这里 upsert 偏好条目。

---

## 二、候选人管理（CandidateLibrary）

业务定位：**简历的结构化档案库**。把非结构化 PDF 转成可检索、可复用的候选人 Profile。

**核心交互**：

- **左栏 · 候选人列表**：按姓名/邮箱搜索，新建候选人，删除候选人；带头像色块与最近更新时间。
- **右栏 · 候选人详情（多 Tab）**：
  - **原始 PDF**：浏览器内嵌预览（带加载/失败回退）。
  - **解析简历**：渲染从 PDF 抽取的结构化字段（教育、技能、经历）。
  - **原文**：纯文本提取结果，便于排查解析问题。
  - **JSON**：完整的 `parsed_json`，对接后续 Agent 推理。
  - **投递记录**：展示该候选人在所有 JD 上的投递历史与状态。
- **后台轮询**：上传 PDF 后自动启动 `parse_status` 轮询（pending → parsing → completed / failed），无需手动刷新。

> 解析能力对应技术报告里的"Qwen API + CoT 任务分解"模块，F1 由 66% → 78%。

## 三、批量化评估（AssessmentDashboard）

业务定位：**流水线作业**。在大量投递中"过一遍粗筛"，输出标准化分数。

**典型工作流**：

1. **筛选投递**：按"企业 / JD"两级下拉过滤投递列表。
2. **批量勾选**：表格头支持全选、单条勾选；顶部实时显示"已选中 N 项"。
3. **一键发起**：两种批量任务并列：
   - **JD-CV 匹配评估**（`jd_cv_match`）：5 个维度 — 技能匹配、经验相关性、教育背景、行业契合度、综合潜力。
   - **简历真实性评估**（`authenticity_check`）：5 个维度 — 时间线连贯性、职责合理性、成就可信度、信息完整性、一致性。
4. **结果展示**：
   - 每行显示双分数徽章（匹配分 + 真实性分），按 80/60/40 分阈值染色（绿 / 黄 / 橙 / 红）。
   - 点击分数即可弹出**评估报告 Modal**：环形分数 + 等级徽章 + 总结 + 维度条形图 + 优势 / 不足（匹配）/ 风险点 / 建议追问（真实性）。
   - 批量结束后顶部显示成功 / 失败计数。

> 这一页的目标是"快"——给出量化结论让 HR 先压缩漏斗，深度判断交给第 4 页。

## 四、深度评估 Agent（AssessmentAssistant）★ 重点

业务定位：**项目的核心交付物**。把"打分流水线"升级为"会调工具、有记忆、能规划"的 Agent，让 HR 能像和资深面试官对话一样深挖每一份候选人或每一个岗位。

页面入口：`/assess/assistant`，源码：`frontend/src/pages/AssessmentCenter.tsx`（同时导出 `AssessmentAssistant`）。

### 4.1 整体布局：三栏 + 顶部模式切换

```
┌─────────────────────────────────────────────────────────────────────┐
│  顶部 page-header：模式切换（投递对话 / 岗位对话） + 显示/隐藏上下文 │
├──────────┬─────────────────────────────────────┬────────────────────┤
│  左栏    │  中栏 · 多轮对话区                   │  右栏 · 上下文 /   │
│  投递 /  │  ─ 会话 Tab 条 (新对话 + 历史会话)   │  Agent 能力面板    │
│  岗位列表│  ─ 消息流 (用户气泡 / Assistant      │  (上下文｜能力     │
│  + 候选  │     气泡 / Plan Tracker / 思考动画)  │   两个 Tab)        │
│  人勾选  │  ─ 输入区 (Plan 切换按钮 + 发送)     │                    │
└──────────┴─────────────────────────────────────┴────────────────────┘
```

侧栏宽度可拖拽（180–500px），左侧栏可一键收起，整个布局充分利用 1080p+ 屏幕。

### 4.2 两种对话模式

| 模式 | 触发条件 | 上下文范围 | 适用场景 |
|------|----------|-----------|----------|
| **投递对话** (`application`) | 选中一条投递记录 | 单个候选人 × 单个 JD | 单点深挖：一个人到底合不合适 |
| **岗位对话** (`jd`) | 选中 JD + 勾选若干候选人 | 一个 JD × N 个候选人 | 横向比较：在 5 个候选人里选谁 |

切换模式会自动清空当前会话与上下文，并显示对应的 Welcome 消息。岗位模式下，**候选人勾选区**支持"全选 / 清空"，会话一旦建立则自动锁定本次讨论的候选人池（避免上下文飘移）。

### 4.3 会话管理（持久化 + 断流恢复）

这是这一页最容易被忽略但工程含量最高的部分：

- **多会话 Tab 条**：每条投递 / 每个 JD 下都有独立的历史会话列表（`listChatSessions`），支持「+ 新对话」、点击切换历史、× 删除。
- **跨页面持久化**：用 `sessionStorage` 同时缓存：
  - `assess_chat_nav`：当前模式 / 选中 ID / `sessionId` / `runId` / `lastSeq`
  - `assess_chat_msgs`：完整消息列表
  即使用户跳到别的页面再回来，对话状态、Plan 步骤、已收 token 都能原样恢复。
- **流式中断恢复**：如果用户在 Agent 还在流式输出时离开页面，回来后会自动调用 `resumeAgentStream(runId, lastSeq)` 从断点续传，不会丢失中间事件；恢复失败则 fallback 到 `listChatMessages` 重新拉取历史。
- **自动滚动**：底部锚点 `msgEndRef` 自动跟随；用户手动向上滚 120px 以上则关闭跟随，避免打断阅读。

### 4.4 SSE 流式事件渲染

后端通过 SSE 推送 7 类事件，前端用 `chatWithAgentStream` 的 `ChatStreamHandlers` 分别接管：

| 事件 | 前端反应 |
|------|----------|
| `onSession` | 拿到 `session_id` / `run_id`，刷新会话 Tab 条 |
| `onEventSeq` | 维护 `lastSeqRef`，用于断流续传定位 |
| `onThinking` | 渲染顶部 *"正在制定执行计划… / 执行步骤 3/7…"* 进度条 |
| `onPlan` | 把 plan 数组渲染为 **PlanTracker** 组件 |
| `onStepDone` | 把对应步骤标记为 ✓，并把下一步置为 *executing* |
| `onReplan` | 保留已 done 的步骤，后续步骤替换为新计划 |
| `onToken` | 把 token 增量 append 到 Markdown 气泡（`react-markdown` + `remark-gfm`）|
| `onDone` / `onError` | 收尾、停止 spinner、错误兜底 |

### 4.5 Planning 模式（Plan-and-Execute 可视化）

输入框右侧的 **Plan** 切换按钮控制 `use_planning` 字段，开启后：

1. 用户消息送出后立即出现一条 **PlanTracker** 气泡（占位、空步骤）。
2. 收到 `onPlan` → 渲染步骤列表，第一步置为 *executing*（带 spinner），其余 *pending*（灰点）。
3. 每收到 `onStepDone` → 该步打 ✓ 并展示一段 `result_preview`，下一步自动接力。
4. 收到 `onReplan` → 保留已完成步骤，后续重排（体现 LangGraph 的动态重规划）。
5. 全部 done 后，最终 Markdown 报告以**新气泡**追加在 PlanTracker 之后。

每个 done 步骤的预览可点击展开成完整 Markdown 详情，支持折叠：

- 进度条（`{done}/{total}` + 百分比填充）
- 三态图标（✓ / spinner / dot）
- 可展开的 `plan-step-detail`，渲染为 Markdown
- 折叠时显示一行 preview 文本

非 Planning 模式则走"普通流式回答"路径，token 直接打到一条 assistant 气泡里。

### 4.6 右侧上下文面板（Context Tab）

根据当前模式动态展示，给 HR 一份"边聊边查"的速览卡片：

**投递对话模式下展示**：

- **候选人**：姓名、邮箱、电话
- **简历摘要**：文件名、解析状态（待解析 / 解析中 / 已解析 / 解析失败）、原文件下载链接
- **岗位信息**：JD 标题、所属企业、JD 描述（截断 300 字）
- **投递状态**：状态徽章 + 最新匹配分 / 真实性分
- **评估报告**：每份历史 `Assessment` 渲染为 **AssessmentReportCard**，支持点击展开各维度小分
- **企业偏好 / 岗位偏好**：展示 `PreferenceMemory` 的 key-value 对，含置信度百分比

**岗位对话模式下展示**：

- 岗位信息卡
- "已加入讨论" 候选人列表（头像 + 姓名 + 状态 + 匹配分）
- 企业偏好 + 岗位偏好

### 4.7 右侧 Agent 能力面板（Capabilities Tab）

通过 `getAgentCapabilities` 拉取后端实际注册的能力清单，分三类透明地展示给用户：

- **技能模块（Skills）**：自研 Skill 名称 + 描述 + 是否包含 *脚本 / 参考文档* 标签。
- **内置工具（Tools）**：传统 Function Calling 工具的 name + description。
- **MCP 服务（mcp_servers）**：每个 MCP server 的 id、传输方式（`transport`）、tag、启用状态指示灯，启用时还会列出该 server 提供的全部 tool 与简介。

> 这相当于一个"Agent 能力说明书"，让 HR 知道 AI 此刻能调哪些外部能力，不再是黑盒。

### 4.8 这一页串起来的技术叙事

把 4.1–4.7 串起来看，正好对应"为什么必须是 Agent"的三大要素：

| Agent 要素 | 在这一页的体现 |
|-----------|---------------|
| **Function Calling** | 能力面板里的 Tools / MCP / Skills；上下文里的简历、JD、KG 偏好都是工具产物 |
| **Memory** | 会话级 Tab 条 + sessionStorage + 断流恢复；上下文面板的"企业 / 岗位偏好"是长期记忆 |
| **Planning** | Plan 切换按钮 + PlanTracker + Replan 事件，把 LangGraph 的多步规划与重规划完整可视化 |

这也是它和第 3 页"批量评估"最本质的区别——**第 3 页只能给一个分数，这一页能讲清楚"为什么是这个分数"，并支持继续追问**。

## 五、Eval 系统（EvalCenter）

业务定位：**面向开发者的离线评测工作台**。不在业务流上，专门用来回归 Agent 的质量。

**三大评测场景**（与后端 `evaluators` 对齐）：

- `jd_cv_match`：JD-CV 匹配
- `authenticity`：真实性检测
- `planning`：Planning 编排

**主要功能区**：

- **左侧 · 创建 Run**：勾选场景、设置 tag（如 `dev` / `prod`）、并发数、case 过滤器；一键 `createEvalRun`。
- **中间 · Run 列表**：所有历史 run 状态徽章（排队 / 运行 / 完成 / 失败）；运行中的 run 自动 4s 轮询刷新进度；支持删除。
- **右侧 · 详情面板（三 Tab）**：
  - **Summary**：
    - `aggregate_scores`：通用指标（如准确率、F1）
    - `business_metrics`：分场景的业务指标（误判率、计划冗余率、图谱回环率等）
  - **Cases**：每条 case 的输入 / 期望 / 实际输出 / evaluator 子分；支持文本搜索定位。
  - **Compare**：选定 baseline run 后做 `compareEvalRuns`，逐指标显示 Δ 与方向（区分"越大越好" vs "越小越好"，自动反色）。
- **数据集同步**：`syncEvalDataset` 一键把当前评测集同步到 LangSmith dataset，便于在 LangSmith 上做更复杂的标注与回放。

> 这一页让"每一次模型 / Prompt / Skill 的改动"都能被定量回归，是 Agent 项目能持续迭代的工程基石。

## 六、串联视角：从一份简历到一次决策

```
候选人管理            企业与岗位管理              批量化评估              深度评估 Agent
─────────             ─────────────              ──────────              ──────────────
PDF 上传  ──┐                                                            ┌── 投递对话：单点深挖
解析 JSON   │       创建企业 → 发布 JD                                    │
档案沉淀    └──→  在 JD 下挂载候选人  ──→  批量打分 (匹配 + 真实性)  ──→ ┴── 岗位对话：横向比较
                                                                              │
                                                                              ↓
                                                                         偏好沉淀回 JD/企业
                                                                              │
                                                                              ↓
                                                                          Eval 系统回归
```

第 1、2 页提供"原料"，第 3 页做"粗筛"，第 4 页做"精评 + 沉淀经验"，第 5 页保证"模型升级不退化"。
五个页面构成一个闭环：**HR 的判断质量越用越高，Agent 的能力也越评越准**。
